# 契约闸门交付:一套可复用的 AI agent 交付流程

*从一套 69-skill 的生产级 agent 系统中提炼,裁剪为技术栈无关的内核。*

## 1. 问题:放任式 agent 交付会漂移

把一个有能力的模型扔给一个任务、让它直接写代码,会出现一个可预测的失败模式:它跳过枯燥的步骤;现实不方便时悄悄改掉需求;把"快好了"四舍五入成"做完了",而你在两步之后——评审、QA 或线上——才发现缺口。这不是恶意,而是乐观 + 缺少强制装置。测试到底有没有真的跑过,模型并不真正在乎。

光靠"更好的提示词"修不了这个问题,因为提示词恰恰是模型可以自由重新解释的那层表面。真正的修法是:**把关键决定从散文里挪出来,交给脚本去判。**

## 2. 思路:契约驱动 + 闸门治理的闭环

契约闸门交付把任何"意图 → 上线"的任务拆成四个阶段,每个阶段以一道闸收口:

```
意图 ─▶ ① 出契约 ─▶ ② 评审 ─▶ ③ 落地 ─▶ ④ 验证 ─▶ 上线
            │           │          │          │
       contract.json  覆盖闸    gate-check   dod-check
                                              │
                       └─────── 红灯 ◀────────┘
```

把上下游连起来的,是 ① 产出、③ 消费的**机器可读契约**。下游需要的一切都在这个文件里,重要的东西不靠从聊天记录里重新推导。设计与实现的解耦,就像模具与铸件的解耦。

## 3. 四个阶段

**① 出契约。** 把意图转成结构化 spec 和一个 `contract.json`(schema 见
[`kit/shared/contract-schema.md`](../kit/shared/contract-schema.md))。契约列出必须覆盖的 `sources`、
被改动的 `entities`,以及每条都**可测试**、且都映射回所满足来源的 `acceptance` 验收项。
**闸门:** `validate_contract.py`——结构完整,且不含任何占位符(`TODO`、`略`、`TBD`…),
"假装完整"正是从这些占位符里溜进来的。

**② 评审。** 在写任何代码之前,检查契约是否忠实覆盖来源、是否可实现。发现"漏需求"最便宜的地方就在这里。
**闸门:** `coverage_gate.py`——每个 `must_cover` 来源都被至少一条验收项引用。名义覆盖但内容空洞的,
由人/agent 评审者降级;有 must_cover 来源没被覆盖,就是硬退回 ①。

**③ 落地。** 对着契约实现——消费它,而不是重新推导它。**闸门(改任何代码之前):**
`gate_check.py`——契约必须是 `reviewed` 且每条验收项可测试。闸门红灯,就**不许动实现代码**。
这是对最诱人的那种漂移的护栏:*"这 bug 一看就懂,我直接改了。"*

**④ 验证。** 完成与否由脚本判,不由叙述判。**闸门:** `dod_check.py`——每条验收项 `passed`、
零未关闭任务、测试套件真的跑过且未被跳过。只要闸门红着,就不许出现任何"完成"措辞。

## 4. 五条不变量

这五条才是让闭环"立得住"而不只是"看着整齐"的东西:

1. **单一事实源。** 规则只写一次,放在 `kit/shared/`。skill 引用它,不复述它。一条规则有了两份拷贝,
   一个月后就成了两条*不同*的规则。
2. **机器可读契约串联。** `contract.json` 是跨越 ①→③ 边界的唯一产物,于是这条边界可检查、可测试。
3. **确定性闸门。** 模型不能自证。一个 `check()` 函数要么非零退出,要么不。这是承重墙。
4. **只编排不复制。** 上层*调用* skill 和工具,绝不重造它们。系统靠"接线"生长,不靠"复制"生长。
5. **派生物不入库。** 索引、缓存、运行态一律 `.gitignore`——它们是可再生的产物,不是源。

## 5. 为什么闸门必须脚本机判

这是整套方法的核心,值得直白说清。"做完之前确保所有测试通过"这种指令,是模型可以绕过去合理化的建议
("这条挂掉的测试看起来跟我没关系")。而一个读契约、除非 `tests.passed && !tests.skipped` 否则退出 1
的脚本,是你能挂 CI 的事实。建议和闸门的区别,就是"指望"和"知道"的区别。

具体地:本 kit 的 `dod_check.py` 约 30 行,拒绝放行任何套件被跳过的契约。仅这一条拒绝,就消除了
"agent 工作悄悄没做完"最常见的那条路径。

## 6. 乐观主义问题与反偷懒

模型默认会上演"完整性表演":一段标着"(篇幅所限,略)",一条验收项因为*大概*做了就标成完成。
kit 里有两个机制对冲它。其一,**禁用占位符**——`validate_contract.py` 让任何含 `TODO`、`略`、`TBD`、
`FIXME` 的契约不通过,于是你无法交一个空壳 spec。其二,**机判 DoD**——完成由证据字段算出,而非断言。
两者都不依赖模型的善意。

## 7. 治理:双轨模式(可选)

一套成熟的 agent 系统终将开始改*自己*——新增 skill、改闸门、调规则。如果这些变更和产品变更走同一条管线,
两者会互相污染。可选的[双轨扩展](../kit/extensions/dual-track-governance.md)把它们分开:一轨"改产品",
一轨"改 kit"。一开始不必上;等到有不止一两个人在动 skill 时再加。

## 8. 适配你自己的技术栈

四个 skill 和四个闸门刻意不含任何语言/框架专有内容。落地方式:把 `kit/` 拷进你的仓库;在
[`kit/shared/rule-sources.md`](../kit/shared/rule-sources.md) 填你团队的真实规则;用你的领域字段扩展契约
schema;把四个脚本接进 CI(规划阶段跑 validate + coverage,构建任务前跑 `gate_check`,合并前跑
`dod_check`)。完整清单见 [`adapting-to-your-stack.md`](./adapting-to-your-stack.md)。

## 9. 局限与何时不要用

这是有成本的。对一行用完即弃的脚本、对明天就删的探针,这套仪式带来的成本大于收益——直接跳过。
当工作要跨人、跨会话共享,当"完成"是有后果的,当你把真正的实现委派给 agent 时,这套 kit 才物有所值。
它也只和喂进契约的证据一样诚实:`dod_check` 信任 `tests.passed` 反映了真实运行。让证据与现实之间的缝隙
尽量小——最好让 CI、而不是模型,来写那些字段。
